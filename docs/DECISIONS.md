> **新しい判断をしたら末尾に追記すること**（消さない）。フォーマット: 日付 / 決定 / 背景 / 選択肢 / 理由 / 影響。

## ADR-001: データは Supabase を直接（ORMなし）
- **決定**: `@supabase/supabase-js` を直接使い、ORMは導入しない。
- **理由**: 小〜中規模で構造が単純。ORMの学習/保守コストを避ける。`lib/supabase-data.ts` に集約して境界を保つ。
- **影響**: SQL/クエリは `lib/supabase-data.ts` に集中。DBカラム変更時は `to*` マッピングも直す必要。

## ADR-002: 画面は単一巨大ファイル `app/page.tsx`
- **決定**: 管理画面・メンバー画面を1つの client component に内包（`AdminDashboard` を `DashboardPage` が内包）。
- **背景**: 段階的に育った結果、約10,700行。
- **理由（現状維持の）**: 分割は大工事でリスクがある。既存の `adminSection` state による切替が機能している。
- **影響/負債**: 保守が重い。**新規機能はできるだけ `lib/` にロジックを出し、将来のコンポーネント抽出に備える**。分割は独立タスクとして計画的に。

## ADR-003: 管理番号は単純連番（登録日は保持しない）
- **決定**: `invoice_number` は `max+1` の連番。`users.created_at` は追加しない（当面）。
- **背景**: 登録日カラムが無く、後付けしても既存メンバーは全員「追加時刻」になり無意味。
- **理由**: 連番は「登録の新しい/古い」の順序は表せる。休眠/未稼働の判別には順序で十分。
- **影響**: 「本当の新人 vs 昔登録して未稼働」を日付で自動判別はできない。未稼働一覧は管理番号降順で緩和。将来 created_at を足すなら「追加後の新規のみ正確」と割り切る。

## ADR-004: 休眠を「休眠」と「未稼働」に分離
- **決定**: 在籍・時給制・非インターンを、①休眠（過去に打刻歴あり・直近3ヶ月なし）②未稼働（一度も打刻なし・新人含む）に分ける。
- **背景**: 「3ヶ月打刻なし」だけだと新規加入直後の人が休眠に誤検出される。
- **理由**: 登録日が無い中で唯一確実な判別軸は「過去に打刻歴があるか」。歴があれば新人ではない。
- **影響**: 休眠カウントに新人が混ざらない。未稼働の一括無効化は新人混在リスクありのため注意書き＋管理番号降順。

## ADR-005: 一括無効化は既存の単体無効化を流用
- **決定**: 休眠/未稼働の一括無効化は `updateMember(id,{isActive:false})` を選択分ループ。
- **理由**: 既存の単体無効化と完全に同経路＝安全・可逆（アーカイブ復元可）。独自実装でロジックがズレるのを避ける。
- **影響**: 破壊的操作なので確認ダイアログで対象氏名を明示必須。

## ADR-006: パフォーマンス改善をフェーズ分け
- **決定**: フェーズ1（初回二重ロード解消＋SELECT列限定）を先に実施。フェーズ2（表示ロードの期間制限A／サーバー集計D／page分割E）は別途設計。
- **理由**: フェーズ1は「クライアントが受け取るデータの中身を変えない」低リスク改善。フェーズ2は履歴系画面（請求/CSV/予実乖離/カスタム集計）の動作確認が要る。
- **影響**: 現状はフェーズ1のみ完了。重さ再燃時はフェーズ2-Aを最優先で設計着手。

## ADR-007: 率(%)の2桁表示は「丸めずに再計算」
- **決定**: 2桁精度が要る所は `safeRatePercent`(1桁丸め) ではなく `ratePercent2`(`Math.round(x*10000)/100`) を使う。
- **背景**: 既存の率計算が1桁丸め。表示だけ `.toFixed(2)` にしても末尾ゼロになるだけ。
- **影響**: `lib/admin-dashboard-metrics.ts` と `app/page.tsx` に `ratePercent2` が点在。統一/共通化は将来の掃除対象。

## ADR-008: 管理者はログイン時から管理者画面
- **決定**: `handleLogin` で admin アカウントなら `isAdminMode=true`。
- **背景**: セッションはリロードで消え毎回ログインするため、ログイン処理を直せば毎回管理画面で始まる。
- **影響**: 右下トグルでメンバー画面へ切替は引き続き可能。

## ADR-009: サイドバーUI化（管理者のみ）
- **決定**: 横タブ→左サイドバー、紺ヘッダー廃止・全幅化。メンバー画面は従来のまま。
- **理由**: セクション切替は既に `adminSection` state 管理なので「ガワ」の差し替えで済み低リスク。
- **影響**: スマホはハンバーガーでドロワー開閉。アイコンは依存追加せずインラインSVG(`AdminNavIcon`)。

## ADR-010: Slack日次結果の土日スキップは「対象日ベース」
- **決定**: `slack-report`（深夜0時・前日実績）で、レポート対象日が土日ならスキップ。
- **理由**: 既存 `slack-daily`（朝・稼働予定）と同方式で統一。金曜実績（土曜0時着）は実データなので送る、土日実績（空）は送らない、が自然。
- **影響**: cronはスキップ、手動POST `test:true` は送信可（検証用）。
